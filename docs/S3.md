# S3

The connector allow the use of S3 endpoint as a file storage system to retrieve and store files.

To do so you need to configure the connector with your S3 credentials, by creating a credential.

```shell
curl -X POST "http://your-connector/private/credentials" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer your-api-token" \
  -d '{
        "type": "s3",
        "content": {
          "accessKeyId": "your-access-key-id",
          "secretAccessKey": "your-secret-access-key",
          "region": "your-region"
        }
      }'
```
The in response you will get the credential id that you will use to configure resource representation.

> Public bucket access can only be used as a provider, not as a storage.

## Chunked Transfer

The connector handles large files efficiently by processing them in chunks rather than loading the entire content into memory at once.

### Download (GET)

When retrieving a file from S3, the connector streams the response body and **concatenates chunks progressively** into a single `Buffer`:

```
S3 Object
   │
   ▼
[chunk 1 – 64 KB] ──┐
[chunk 2 – 64 KB] ──┤  Buffer.concat()  ──▶  Complete Buffer
[chunk 3 – 64 KB] ──┤
[chunk N – ...]  ───┘
```

- Each chunk is collected as it arrives from the S3 stream.
- The final `Buffer` is only assembled once the stream emits the `end` event.
- This guarantees **byte-perfect integrity**: no byte is dropped, duplicated, or reordered regardless of the file size.

### Upload (POST)

When storing a file to S3, the connector uses the **AWS SDK multipart `Upload`** utility, which automatically splits the payload into parts and uploads them in parallel:

```
Input (object | Buffer | Readable stream)
   │
   ▼
   Readable stream
   │
   ├── Part 1 (≤ 50 MB) ──┐
   ├── Part 2 (≤ 50 MB) ──┤  S3 Multipart Upload (4 concurrent parts)
   ├── Part 3 (≤ 50 MB) ──┤
   └── Part N (≤ 50 MB) ──┘
         │
         ▼
      S3 Object
```

| Parameter | Value | Description |
|---|---|---|
| `partSize` | 50 MB | Maximum size of each individual part |
| `queueSize` | 4 | Number of parts uploaded concurrently |

The connector accepts three types of input data for uploads:

| Input type | Behaviour |
|---|---|
| Plain object / JSON | Serialised to JSON string, then wrapped in a `Readable` stream |
| `Buffer` | Wrapped directly into a `Readable` stream |
| `Readable` stream | Passed as-is to the multipart upload — **no extra memory copy** |

> **Tip:** For very large files, prefer passing a `Readable` stream as the data payload. This avoids buffering the entire content in memory before the upload starts.

### File naming

When uploading, the connector resolves the destination key as follows:

1. If `providerData.fileName` is set on the data exchange, it is used as-is.
2. Otherwise, a **UUID-based filename** is auto-generated (e.g. `3f2a1b4c-….json`).

The final S3 key is: `<key-prefix-from-url>/<fileName>`

### Error handling

| Situation | Error thrown |
|---|---|
| Credential not found | `S3 credentials not found` |
| Credential found but `accessKeyId` or `secretAccessKey` missing | `Missing S3 credential content` |
